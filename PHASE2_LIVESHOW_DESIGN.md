## Owner-locked spec
- Host launches a review song as a live match; HOST PLAYS EVEN IF NOBODY JOINS (non-negotiable).
- Submitter auto-notified on the site + Join-the-match button -> head-to-head.
- Others spectate IN-GAME (both players: live notes + scores).
- Normal MP byte-identical when show mode off. Stability first. softPresence only. Brand black/crimson/chrome/gold.

## SEAM FINDINGS (verified file:line facts)

### answers

- **1. SOLO START: can a room host fire room-start/beginMatch with only 1 occupied seat today? What is the MINIMAL change for 'host plays solo, others watch/join later'?**
  - ANSWER: NO — solo start is triple-gated. (1) The READY button is disabled without an opponent: refreshReadyEnabled at multiplayer.js:662-664 sets `ok = oppPresent && !!sel.trackId` and `rb.disabled = !ok`, and oppReady only ever flips via the opponent's 'ready' broadcast (onReady:682). (2) maybeStart's room branch at :696 requires `room.id && room.isHost && !matchCh && meReady && oppReady && sel.trackId` before calling startRoomMatch. (3) startRoomMatch itself hard-guards `if (!room.isHost || !room.p2) return;` at :1517-1518. The only solo path today is 'Add a CPU' (:3356-3362) which CLOSES the room (`closeRoom(true)`) and runs devVsNpc — killing any spectators. Note devVsNpc (:3192-3223) is an existence proof that beginMatch runs fine with matchRole='host', an inert channel, and no ready handshake: it sets matchCh to a fake channel, sets sel, and calls `beginMatch(Date.now()+VS_LEADIN_MS, sel)` directly. MINIMAL CHANGE: (a) relax refreshReadyEnabled + maybeStart's oppReady gate when `room.isHost && !room.p2` in a show room; (b) drop startRoomMatch's `!room.p2` guard for that mode and broadcast `room-start {mid, p1Id, p2Id:null}` — onRoomStart (:1523-1531) already routes everyone who isn't p1/p2 to spectateMatch(mid,...), so spectators work unchanged; (c) host opens the real rr-match-<mid> channel and immediately broadcasts 'start' {atMs, sel, combat} + beginMatch (no ready wait); (d) solo settle: settleIfReady (:1305-1313) with no oppFinal waits on the 8s _settleSafetyT (:1292) then showWinner declares 'YOU WIN' via the `!op → win=true` branch (:1328) — a show wants that replaced with a return-to-room-waiting instead of a hollow victory screen. Also mind the start-watchdog inside resolveAndStart (:1152-1166): if #game/#loading never activates past atMs+4s it calls abortRound back to setup.
  - IMPLICATIONS: Solo-preview is a small, surgical delta (3 gate relaxations + a solo-settle path), not a new mode. Reuse room-start with p2Id:null so the existing spectator routing at :1527 works for free. Budget for the 10s VS_LEADIN_MS (:92) countdown and keep the watchdog happy (the review launch card must actually reach #loading/#game). Never touch these gates when the show flag is off — normal MP must stay byte-identical.
- **2. ROOM-START PAYLOAD: exactly what does the host broadcast when a room match starts, and where does each receiver consume it (guest vs spec)?**
  - ANSWER: TWO broadcasts on TWO channels. (A) Room channel 'room-start' payload = `{mid, p1Id, p2Id}` ONLY — no sel, no atMs (startRoomMatch:1519-1521). Consumed by onRoomStart (:1523-1531): spectator path — `if (room.seat==='spec' || (ME.id!==p.p1Id && ME.id!==p.p2Id)) spectateMatch(p.mid, members[p1Id], members[p2Id])` (:1527); seated path — role = p1?'host':'guest', `_fromRoom = {sel: …}` carries the host's locked pick (:1529), then startMatchChannel(p.mid, role, oppMeta) (:1530). (B) Match channel 'start' payload = `{atMs, sel, combat}` (maybeStart:698-700), where sel = {trackId,title,artist,art,difficulty,demo,env} — consumed by the guest's onStart (:722) → `matchCombat=!!p.combat; beginMatch(p.atMs, p.sel||sel)`. The room→match handoff auto-fires it: enterSetup's _fromRoom block (:569-581) re-broadcasts 'song' and auto-READYs both sides ~350ms after channel subscribe, so maybeStart's matchCh branch (:697-720) emits 'start'. Supporting match-channel events: 'song' {sel} (broadcastSong:613 — also mirrored on the ROOM channel pre-start, handler at :1498), 'ready' {ready,id} (:673), 'round' {rid} (:716), 'tick' {score,combo,acc,prog,name} ~6/s (:1215), 'state' = full getRenderFrame {sc,cb,mu,od,oda,st,pr,ev[]} ~13/s (:1240), 'final' {name,score,combo,acc,grade} (:1271), 'rematch' {} (:3365), 'shock' {from,combo,od} (:1229). Spectators register handlers ONLY for 'tick' and 'final' (spectateMatch:1786-1787) — they never consume 'start' (so no sel/atMs) and never consume 'state'.
  - IMPLICATIONS: The live-show catch-up problem is precisely that sel+atMs live only in the one-shot match-channel 'start'. Cheapest fix: have the host repeat a compact `show-snapshot {mid, sel, atMs, combat, state}` on the ROOM channel every ~4s (copy the t-snapshot pattern at :1951-1964) — then room-start needs no schema change and every late arrival self-heals. Spectators should also subscribe 'start' and 'state' on the match channel.
- **3. SPECTATOR SEATS: how does joinRoom(rid, asSpec=true) work, what do spectators get at room-start, and what happens to one who arrives AFTER the match started?**
  - ANSWER: joinRoom(rid, true) at :1534-1543 sets `room.seat='spec'`, `spectating=true`, `room.p2=null`, and joins rr-room-<rid> via joinRoomChannel (:1491-1503) with softPresence meta {id,name,avatar,seat:'spec'}. There is NO spectator count limit in code — only the PLAYER gate `!asSpec && meta.count >= meta.max` (:1537); the practical ceiling is the softPresence ≤10-peers/channel design (:155-161). Host-side seat bookkeeping ignores specs: onRoomPeers (:1505-1515) assigns p2 = first member whose seat !== 'spec'; paintRoomWaiting counts specs for the 'N watching' chip (:1577-1578). At room-start a PRESENT spectator gets the room-channel 'room-start' {mid,p1Id,p2Id} and routes to spectateMatch(mid) (:1527 → :1780-1798): joins rr-match-<mid> (no softPresence — players never see them on the match channel), mounts the compact mp-opp score panel only, listens 'tick'+'final'; the first 'final' fires showWinner with myFinal={score:0} (:1787) → the spectator sees 'YOU LOSE' (guarded from ladder damage at :1334, but the verdict UI is still wrong). A spectator arriving AFTER the match started joins the room channel fine but is stranded at 'Watching this room — the match starts when the host begins' (:1590) — room-start is never re-broadcast, so they never learn the mid. What's missing for catch-up: (a) the mid (the only key to rr-match-<mid>); (b) sel — nuance: a spectator present BEFORE start does receive sel via the room-channel 'song' handler (:1498 → onSong sets module sel), but a late one missed those broadcasts; (c) atMs/progress so they can align a local chart; (d) combat flag. So yes — a periodic room-channel snapshot carrying {mid, sel, atMs, combat} is exactly the whole gap.
  - IMPLICATIONS: Mid-show drop-in spectating = one host-side heartbeat + a late-join handler that calls spectateMatch(mid) with the snapshot's sel/atMs. Also fix the spectator verdict (suppress showWinner's YOU-LOSE for spectating=true, wait for BOTH finals keyed by name/id) and keep spectators off match-channel softPresence as today to preserve the ≤10-peer budget.
- **4. LATE GUEST JOIN: if the 2nd player joins the room while the host is mid-run, what happens today? What would 'join as challenger for the NEXT run' need?**
  - ANSWER: Today they can't even get a seat: during the match the host keeps BOTH the room channel (never left — startMatchChannel doesn't call leaveRoomChannel, roomSP keeps beating) and the lobby channel (syncActive:3609 preserves lobby while `matchLive||matchCh`), so room-ping (:391) still answers with advertiseRoom count = `1 + (room.p2?1:0)` = 2/2 (:1478-1480) → joinRoom's gate (:1537) bounces the joiner to 'Room is full — spectate instead.' If a seat WERE free (solo-show host), the joiner takes p2, the host's onRoomPeers auto-assigns room.p2 (:1509-1513) even mid-song, and the joiner then sits at 'Waiting for the host to start…' (:1592) with zero knowledge of the live run. A NEXT-run challenger flow needs: (1) room-meta to advertise live-state (add `live:true` to the :1479 payload) so the browser/JOIN button can say 'match in progress — join the next run'; (2) the room-channel snapshot from Q2/Q3 so the joiner's client knows a run is live (paint 'up next' instead of 'waiting for host'); (3) host-side on song end: instead of only stepping to 'winner', re-enter room waiting (enterRoomWaiting:1556) and fire startRoomMatch with the NEW room.p2 — note the existing rematch path ('rematch' event, :3365 → resetForRematch:1346) rides the OLD matchCh between the ORIGINAL two peers, so a new challenger requires teardownMatch + a fresh room-start with a fresh mid; (4) beware onMatchPeers (:597-601): while matchLive, an absent opponent triggers markOppGone + settleIfReady — a solo host with no matchSP peers must not trip that.
  - IMPLICATIONS: The submitter's 'Join the match' button should deep-link into the ROOM (not the match channel) and take the p2 seat; the host's client then ends/finishes the solo run and relaunches via a fresh room-start — giving the head-to-head on the same review song with all existing sync machinery. Keep room.p2 auto-assignment as the join mechanism; just add the live-state signaling and the post-run relaunch.
- **5. MATCH END → ROOM: after showWinner/settle, do players return to the room? Can the host relaunch the same song for a rematch?**
  - ANSWER: Players land on the MP 'winner' step (showWinner:1314-1343 re-adds .active at :1341 since the engine's results screen stripped it). Rematch works TODAY and reuses the same song: #mpx-rematch (:3365) broadcasts 'rematch' on the STILL-OPEN matchCh (handler :539) and both sides run resetForRematch (:1346-1361) → step('setup') with matchCh, matchId, room.id, room.isHost, and sel all preserved — the host is still amPicker() (:81, room.isHost true), can keep or re-pick the track (picker :3373, broadcastSong hits matchCh :613), both READY → maybeStart's matchCh branch (:697) fires a fresh 'start' {atMs, sel} and a fresh server round (_roundStartFired cleared at :1349). The ROOM channel survives the whole match (never unsubscribed), so the room, its spectators, and its directory entry persist. CAVEATS: (a) resetForRematch does NOT call enterRoomWaiting, so the room strip/invite bar stays hidden (enterSetup hid it at :552-554) — cosmetic; (b) #mpx-backlobby (:3364 → backToLobby:1378-1382) tears down the MATCH but NOT the room — room.id/room.ch/roomSP linger as a zombie the host is still advertising (only #mpx-leave-win → leaveAll (:3363, :355-367), #mpx-room-close (:3344), or mp-back (:3595) actually closeRoom); (c) the rematch stays between the ORIGINAL matchCh peers — a spectator promoted to challenger needs a new room-start/mid (see Q4).
  - IMPLICATIONS: For the show loop (host previews → submitter joins → head-to-head → maybe re-run), the 'winner→setup→READY→start' cycle is already solid for same-pair reruns on the same review song. Design the challenger-swap as room-level relaunch, not matchCh 'rematch'. Fix or avoid the backToLobby zombie-room path in show mode.
- **6. sel CONSTRUCTION: where does the host build sel for a 1v1, and where would a review-track sel injection be cleanest?**
  - ANSWER: sel is built in exactly one place for humans: renderPicker's row click handler (:3275-3283) — `sel.trackId=t.id; sel.title; sel.artist=t.artist_credit_name||t.artist_name; sel.art=t.artwork_url; sel.demo=(t.id==='demo')` from `RhythmCatalog.allTracks()` filtered by trackReady + !isVideo (:3264-3266); difficulty via #mpx-diff (:3367-3372); stage via the chip row (:645); then broadcastSong (:613). The choke point for a review track is resolveAndStart:1110 — `RC.allTracks().filter(x=>x.id===s.trackId)[0]` → a review track is NOT in catalogTracks (catalog.js:1010) → `!t` → demoProvider (:1114-1115). sel carries NO audio URL, so peers can't resolve it either (and /review/resolve is host-locked, so guests/spectators can't fetch it themselves). CLEANEST INJECTION (two small seams): (1) multiplayer.js resolveAndStart — before the demo fallback, honor a new optional `sel.audioUrl`: `if (!t && s.audioUrl && !/\.m3u8(\?|$)/i.test(s.audioUrl) && RhythmGame.__buffered) prov = () => RhythmGame.__buffered(s.audioUrl, {title:s.title, artist:s.artist, artwork:s.art})` — one branch, additive, and it transports to guest AND spectator for free because sel rides 'song'/'start'/(future snapshot); (2) catalog.js — the review card already holds the full synthetic track with the decodable analysis_url (_resolveReview:1888-1893, `audio_url: aurl`); add a 'START LIVE MATCH' action next to rvl-play (_openReviewLaunch:1833-1845) that calls a new small entry, e.g. RhythmMP.openShowRoom({trackId, title, artist, art, audioUrl, difficulty}) which opens a room (openRoom:1465 pattern) with sel pre-seeded and broadcastSong'd. Alternative (even less MP churn): have catalog.js push the synthetic review track into catalogTracks so the :1110 lookup just works — but that pollutes the library UI and STILL doesn't resolve for peers (their allTracks() lacks it), so the sel.audioUrl seam is the correct one for peers regardless.
  - IMPLICATIONS: Plan on both edits: sel.audioUrl in resolveAndStart is the load-bearing change (peers + spectators resolve the same decodable .m4a → identical deterministic chart), and the review card is the natural host-side entry. The analysis_url is the same visibility class as catalog audio_url (already public via /tracks), so broadcasting it on the room/match channel is not a new exposure. Keep sel.trackId = the real track uuid so game_plays/mp settle rows stay attributable (game_plays has no FK, accepts it).

- **8. getRenderFrame/getGhostNotes/renderGhost + couch renderP2Deck: what would a SPECTATOR need, concretely, to render TWO live note-highways?**
  - ANSWER: Building blocks (all public): game.js:6551 getRenderFrame() → {sc,cb,mu,od,oda,st,pr,ev[]} (DRAINS _rfHits — call once/frame, mp:1236-1240 caches as _myRf); game.js:6561 getGhostNotes(aheadSec) → pooled {n, items:[{lane,d,type 0tap/1hold/2chord/3bomb}]} read from the LOCAL notes[]+songTime() (96-note pool, uncut by your own hits per :6568-6572); game.js:2151 getLaneFrame() → {nearX[],farX[],nearY,farY,lw,persp,warp,colors[]} lane geometry; game.js:2166 getGuitarArt() → warped board art; game.js:6590 startAt(prov,{atMs,difficulty}) + :6528 __buffered(url,meta) for the deterministic chart. Consumers to copy: multiplayer.js:931 renderGhost (single rival deck: chart from getGhostNotes + per-player flashes/sparkles from state.ev, catcher-press synth at :1024 when no ev) and couch.js:285 renderP2Deck — the best template because it's a self-contained pure-ish function of (canvas, laneFrame, ghostNotes, eventFrame) already proven to run BESIDE the P1 deck; note both keep module-level _spk/_gFlash pools, so a two-deck spectator needs per-deck pool instances (couch.js shows the pattern). CONCRETE SPECTATOR FLOW: (1) receive {sel(+audioUrl), atMs, difficulty} from the show snapshot; (2) run RhythmGame.startAt(__buffered(sel.audioUrl, meta), {atMs, difficulty}) with music volume 0 (applySettings music:0, game.js:2181) and input ignored — decoding the same audio at the same difficulty yields the IDENTICAL chart, and songTime() locks to the shared atMs, so ONE local engine feeds BOTH decks' note-fall via getGhostNotes/getLaneFrame (both players play the same chart); (3) subscribe the match channel 'state' + 'tick' and key them by player — BLOCKER: 1v1 'tick' (:1215) and 'state' (:1240) carry NO id (tournament t-state stamps `_myRf.id = ME.id` at :2472 and t-tick has id at :2447) — add `id: ME.id` to both 1v1 payloads (receivers ignore unknown fields; additive-safe); (4) per frame, run two renderP2Deck-style draws, deck A fed by player-1's latest state (ev → flashes/sparkles, sc/cb/od → HUD) and deck B by player-2's; (5) suppress the local results/scoring surfaces at song end (the engine will fire endGame/results — spectator mode must swallow it, and must not recordLocal/mpSettle — the spectating gate already exists at :1277/:1334).
  - IMPLICATIONS: The rich dual-highway spectator is genuinely buildable from existing parts: ~1 new file cloning couch.js's deck renderer ×2 + the id-stamping one-liners in startTick. The hard 20% is running the engine as a silent, input-dead, results-suppressed chart clock — that needs a small engine flag (e.g. spectator:true on startAt) rather than hacks, and it must be strip-safe/inert for normal play. Mid-join spectators also need atMs from the snapshot to start the chart clock mid-song (startAt only schedules forward — seeking into a decoded buffer mid-song needs a small __buffered offset param or accepting join-at-next-run).

### surprises

- backToLobby (multiplayer.js:1378-1382) does NOT close the room — after a room match, 'BACK TO LOBBY' leaves room.id/room.ch/roomSP alive and the host still answers room-ping with a 2/2 advert (zombie room). Only mpx-leave-win/mpx-room-close/mp-back call closeRoom. A show flow that programmatically routes the host must use the right exit.
- Spectators present in the waiting room BEFORE start DO receive the host's sel — the room channel has a 'song' handler (:1498 → onSong) that updates module sel live. spectateMatch just ignores it. So the pre-start spectator catch-up gap is smaller than assumed; only LATE arrivals truly lack sel — the mid+atMs is the universal missing piece.
- devVsNpc (:3192-3223) is a working blueprint for solo-start: it sets matchRole='host', an inert matchCh, sel, and calls beginMatch directly with zero ready handshake — proof the engine/HUD path doesn't require an opponent. A show solo-start is this pattern with a REAL channel and the vs-HUD mount gated off.
- A solo run would 'settle' today: with no oppFinal and oppLeft=false, the 8s _settleSafetyT (:1292) force-fires settleIfReady and showWinner's `!op → win=true` (:1328) declares YOU WIN — so nothing hangs, but the host gets a fake victory screen the show design must replace.
- spectateMatch joins rr-match-<mid> WITHOUT softPresence (:1784-1785) — spectators are invisible to players on the match channel (good for the ≤10-peer budget) and the spec count only exists room-side (:1577). Show viewers can therefore scale past the room-roster limit only if they stay presence-silent like this.
- The start-watchdog inside resolveAndStart (:1152-1166) applies to 1v1 too: if neither #loading nor #game is active past atMs+4s it abortRound()s back to setup. A show host lingering on a DOM overlay (e.g. a 'waiting for challenger' card) past the synced start would be yanked — the solo flow must reach the loading screen or disarm the watchdog.
- onMatchPeers (:597-601) auto-settles when the opponent's presence drops mid-match (markOppGone + settleIfReady). In a solo-show match with an empty matchSP roster this never fires (no wasPresent), but the moment a challenger joins-then-refreshes, the host's run force-settles — the plain 1v1 room has NONE of the tournament's reconnect grace (t-snapshot/sessionStorage/isRecentlyAlive), confirming the resilience gap matters even for 2-person shows.
- VS_LEADIN_MS is now 10 seconds (:92, owner-set in build100t) — every live-match start has a 10s cinematic countdown; the show pacing (host clicks PLAY on the review card → 10s to first note) should be designed around it rather than re-shortened.
- getGhostNotes deliberately does NOT cull notes the local player already hit (game.js:6568-6572) — the ghost chart is a full independent note-fall, which is exactly what a spectator's dual-deck needs; no engine change required for that part.
- advertiseRoom hardcodes max:2 (:1480) and roomsDir joins gate on count>=max (:1537) — a show room that wants 'submitter seat + open spectating' advertised on the site needs its own live/count semantics, since the in-game browser will render it FULL the entire show.

### risks

- Notification latency: useNotifications polls every 30s (verified; realtime deliberately removed) — worst-case the submitter sees the invite 30s late. Mitigated by the toast + optional 10s fast-poll (instruction 5) and the host verbally cueing on stream; do not re-enable realtime on notifications.
- Shared-token audience widens preview_only reach: handleTrack's comment (:332-336) says preview submissions 'must never leak to the public — only admins or the host'. A 'game-review-shared' token lets ANY authenticated user resolve that submission's audio for 2h. Exposure is equivalent to the audio already broadcast on the MP room channel (game-side sel.audioUrl seam) and the same visibility class as catalog audio_url, and v1 places the token only in the submitter's private notification row — but it is a deliberate policy delta the owner should ack.
- Anyone can open ?mproom=<rid> today (existing deep link) — the spectate chip does not add new backend exposure, but CHALLENGER SEAT control (only the invited submitter takes p2) is a game-side responsibility; the backend cannot enforce it. The share token is a resolve credential, not a seat credential.
- stream_status.live_match writes are keyed by submissions.host_id; owner-run shows may have host_id null — endpoint needs the RR_OWNER_HOST_ID env fallback (mirrors the site's OWNER_HOST_ID constant) or announce degrades to announced:false. Verify the env is set before flipping announce:true.
- radio_plays credit inside handlePlays targets track_id — when the review resolve fell back to a synthetic submission uuid (no radio_track row yet) that insert silently no-ops/fails-best-effort (verified wrapped, response still 201); play counts for fresh submissions under-credit until Mux ingest creates the row. Cosmetic, pre-existing from Phase 1.
- Ordering: ship endpoint (instructions 1-2) BEFORE the game's START LIVE MATCH button goes live, and NotificationCenter CTA (instruction 4) before the first real on-air use — though the type renders harmlessly (Bell icon, title+message) if sequencing slips, verified via the typeIcons fallback.
- Token TTL: share token inherits the host token's remaining exp — a match started near the 2h boundary can strand late joiners with token_expired; the host's fix is re-opening the review (fresh mint) and re-inviting. Acceptable v1; noting rather than auto-extending keeps the security story simple.

- SHARE TOKEN BECOMES PUBLIC + IRREVOCABLE: the design broadcasts sel.shareTok in 'song'/'start'/show-snap on rr-room-<rid> and rr-match channels. Supabase Realtime broadcast is joinable with the PUBLIC anon key (window.RHYTHM_CONFIG), and rids are advertised on the public rr-lobby via room-meta — so the share token is world-readable the instant the show starts, and stateless HMAC means it cannot be revoked for up to 2h of preview-only submission audio. The 'any authenticated user' gate on the shared-audience /review/resolve is theater. Fix: NEVER put shareTok in any Realtime broadcast — broadcast only sel.audioUrl (the analysis_url the owner already accepted as catalog-visibility class); deliver the share token ONLY inside the submitter's private notification room_url; and hard-restrict the 'game-review-shared' resolve branch to role ∈ {host, submitter} (403 for anyone else — resolves the design's own open question 1 in the restrictive direction). Spectators stay fully tokenless via the MP-channel audioUrl seam.
